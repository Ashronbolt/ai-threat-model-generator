STRIDE Threat Analysis
[Critical] Elevation of Privilege — JWT token with weak signing secret allows attackers to forge tokens and access any patient record
Mitigation: Enforce RS256 asymmetric signing for JWTs, rotate signing keys regularly, and validate token claims server-side on every request.
[High] Spoofing — Attacker impersonates a legitimate patient by replaying a stolen JWT token captured over an unencrypted connection
Mitigation: Enforce HTTPS/TLS across all endpoints, set short JWT expiry (15 mins), and implement token revocation via a Redis blocklist.
[High] Information Disclosure — S3 bucket misconfiguration exposes patient documents publicly due to overly permissive bucket policy
Mitigation: Set S3 bucket to private, use pre-signed URLs with short expiry for document access, and enable S3 Block Public Access at account level.
[High] Tampering — Lack of input validation on the REST API allows SQL injection attacks targeting the PostgreSQL database
Mitigation: Use parameterised queries or an ORM, implement a WAF on the Application Load Balancer, and run regular DAST scans.
[Medium] Denial of Service — Unauthenticated endpoints on the REST API are vulnerable to request flooding, taking down the booking service
Mitigation: Implement rate limiting (e.g. AWS WAF rate rules), add CAPTCHA on public-facing forms, and configure auto-scaling on the load balancer.
[Medium] Repudiation — No centralised audit logging means malicious actions by staff or attackers cannot be traced back to a user
Mitigation: Implement structured audit logs (user ID, action, timestamp, IP) stored in an immutable log service (e.g. AWS CloudTrail + CloudWatch Logs).
[Low] Information Disclosure — Verbose API error messages expose internal stack traces and database schema details to unauthenticated users
Mitigation: Implement a global error handler that returns generic error messages in production; log full errors server-side only.
